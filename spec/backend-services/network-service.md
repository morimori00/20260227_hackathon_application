# ネットワークサービス設計

## 責務
口座間の取引ネットワークグラフデータを構築し、パターン検出結果と合わせて返却する。

## 依存
- DataStore（データローダーサービス）

## エンドポイント実装

### GET /api/v1/network

#### アルゴリズム: BFS（幅優先探索）によるNホップ展開

1. 起点口座ID を受け取る
2. 日付フィルターを適用した取引データのサブセットを取得
3. BFS で N ホップ以内の関連口座・取引を収集:

```
探索キュー = [起点口座]
訪問済み = {起点口座}
ホップ = 0

while ホップ < max_hops and キューが空でない:
    次のキュー = []
    for 口座 in 探索キュー:
        口座が関わる取引を検索（from_account または to_account）
        for 取引 in 関連取引:
            相手口座 = 取引の相手方
            エッジリストに取引を追加
            if 相手口座 not in 訪問済み:
                訪問済みに追加
                次のキューに追加
                ノードリストに相手口座を追加
    探索キュー = 次のキュー
    ホップ += 1
```

4. ノード情報の構築:
   - 各口座の合計取引金額、取引件数を集計
   - account_lookup から銀行名・エンティティ名を取得
   - その口座に関連する取引の fraud_score の最大値をノードの fraud_score とする
   - 起点口座には `is_origin: true` を設定

5. エッジ情報の構築:
   - 各取引について送金元→送金先の方向付きエッジを生成
   - fraud_score, prediction を付与

6. パターン情報の付与:
   - DataStore.patterns の中から、ノードリストに含まれる口座が関わるパターンを抽出
   - 各パターンの種別、詳細、関連取引ID、合計金額を返却

#### パフォーマンス制約
- ノード数上限: 200（超過した場合は fraud_score の高いものを優先して返却）
- エッジ数上限: 1000
- 上限超過時はレスポンスに `truncated: true` を付与
