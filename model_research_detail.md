# Comparative Study of Machine Learning Classifiers for Anti-Money Laundering Transaction Detection

## Abstract

This study investigates the application of machine learning classifiers to detect money laundering transactions within a synthetic financial ecosystem. Using IBM's publicly available Anti-Money Laundering (AML) dataset (HI-Small variant), we trained and evaluated six classifiers: Logistic Regression, Random Forest, Decision Tree, K-Nearest Neighbors (KNN), Support Vector Machine (SVM), and XGBoost. The dataset exhibits extreme class imbalance, with only 5,177 laundering transactions out of 5,078,336 total (0.10%). To mitigate this imbalance, we applied Random Under-Sampling prior to model training. Among all models evaluated, XGBoost achieved the highest F1 score of 0.8892 for the laundering class, though Logistic Regression performed competitively at 0.8825. We further demonstrate that without resampling, even XGBoost yields a recall of only 0.06 for laundering detection, underscoring the critical importance of class imbalance handling in AML applications.

## 1. Introduction

Money laundering is a multi-billion dollar problem in the global financial system. Detection of laundering transactions remains extremely difficult due to the deliberate concealment strategies employed by criminals, the rarity of illicit transactions relative to legitimate ones, and the limited availability of labeled data for model training. Most automated detection algorithms suffer from high false positive rates (legitimate transactions incorrectly flagged) and high false negative rates (undetected laundering).

Access to real financial transaction data is highly restricted for both proprietary and privacy reasons. Even when data access is possible, accurate labeling of transactions as laundering or legitimate is inherently problematic. To address these challenges, we utilize IBM's synthetic transaction dataset, which models an entire financial ecosystem including the full money laundering cycle: placement (injection of illicit funds), layering (mixing into the financial system), and integration (spending the illicit funds).

This study makes the following contributions:

1. A comprehensive exploratory data analysis (EDA) of the HI-Small AML dataset, characterizing the statistical properties and distributional patterns of laundering transactions.
2. A systematic comparison of six machine learning classifiers under identical preprocessing and evaluation conditions.
3. An empirical demonstration of the impact of class imbalance handling on model performance in the AML domain.

## 2. Dataset Description

### 2.1 Data Source

The dataset used in this study is the **HI-Small** (High Illicit ratio, Small scale) variant from the IBM Transactions for Anti-Money Laundering dataset, version 8, available on Kaggle (Altman, 2023). The dataset is generated by IBM's synthetic financial data generator, which models a virtual world of individuals, companies, and banks engaging in financial transactions including purchases, salary payments, loan repayments, and other activities.

The "HI" designation indicates a relatively higher illicit ratio compared to the "LI" (Low Illicit) counterpart. The "Small" designation refers to the scale of the dataset within the HI group, which also includes Medium and Large variants. According to the dataset documentation, the HI-Small dataset covers approximately 515,000 bank accounts.

### 2.2 Raw Data Schema

The original CSV file (`HI-Small_Trans.csv`) contains **5,078,345 rows** and **11 columns** with the following schema:

| Column | Data Type | Description |
|--------|-----------|-------------|
| Timestamp | string | Transaction timestamp (format: YYYY/MM/DD HH:MM) |
| From Bank | int64 | Sender bank identifier |
| Account | string | Sender account identifier (hexadecimal) |
| To Bank | int64 | Receiver bank identifier |
| Account.1 | string | Receiver account identifier (hexadecimal) |
| Amount Received | float64 | Amount received in the receiving currency |

| Amount Paid | float64 | Amount paid in the payment currency |
| Payment Currency | string | Currency of the payment |
| Payment Format | string | Payment method category |
| Is Laundering | int64 | Binary label (0 = legitimate, 1 = laundering) |

## 3. Methodology

### 3.1 Feature Engineering

From the original 11 columns, the following preprocessing was applied:

1. **Timestamp decomposition**: The `Timestamp` column was converted to datetime and a `Week` feature (day of the week name, e.g., "Thursday") was extracted.
2. **Column removal**: `Timestamp`, `Date`, and `Time` columns were dropped after feature extraction.
3. **Type conversion**: `From Bank` and `To Bank` were converted from integer to string (categorical) type.

The final feature set used for modeling consists of **10 columns**:

| Feature | Role in Pipeline |
|---------|-----------------|
| From Bank | One-Hot Encoded |
| Account | One-Hot Encoded |
| To Bank | One-Hot Encoded |
| Account.1 | Dropped (not in encoder list) |
| Amount Received | Dropped (not in encoder list) |
| Receiving Currency | One-Hot Encoded |
| Amount Paid | Dropped (not in encoder list) |
| Payment Currency | One-Hot Encoded |
| Payment Format | One-Hot Encoded |
| Week | One-Hot Encoded |

**Important note**: The `ColumnTransformer` was configured with only 7 categorical columns for One-Hot Encoding and the default `remainder='drop'` setting. As a result, `Account.1`, `Amount Received`, and `Amount Paid` were **implicitly excluded** from the model. The models therefore rely entirely on categorical features (bank identifiers, account identifiers, currency types, payment format, and day of the week) without using any numerical transaction amount information.

### 3.2 Class Imbalance Handling

Given the extreme imbalance ratio of approximately 981:1, Random Under-Sampling was applied using `imblearn.under_sampling.RandomUnderSampler` (random_state=42). This reduced the majority class (normal transactions) to match the minority class (laundering transactions), producing a balanced dataset of **10,354 samples** (5,177 per class).

### 3.3 Train-Test Split

The balanced dataset was split into training and test sets using `sklearn.model_selection.train_test_split` with the following parameters:

| Parameter | Value |
|-----------|-------|
| Test size | 0.3 (30%) |
| Random state | 42 |
| Training samples | 7,247 |
| Test samples | 3,107 |
| Training class distribution | ~50/50 balanced |
| Test class distribution | 1,586 Normal / 1,521 Laundering |

### 3.4 Preprocessing Pipeline

All models share an identical preprocessing pipeline implemented via `sklearn.pipeline.Pipeline` and `sklearn.compose.ColumnTransformer`:

1. **OneHotEncoder** applied to 7 categorical columns with `drop='first'` (to avoid multicollinearity) and `handle_unknown='ignore'` (to handle unseen categories at inference time as all-zero vectors).
2. The encoder output is fed directly to each classifier within a unified Pipeline object.

### 3.5 Models Evaluated

Six classifiers were evaluated, along with a previously saved XGBoost model:

| Model | Implementation | Key Hyperparameters |
|-------|---------------|-------------------|
| Logistic Regression | `sklearn.linear_model.LogisticRegression` | max_iter=2000, random_state=42 |
| Random Forest | `sklearn.ensemble.RandomForestClassifier` | n_estimators=100, random_state=42 |
| Decision Tree | `sklearn.tree.DecisionTreeClassifier` | random_state=42 |
| KNN (k=5) | `sklearn.neighbors.KNeighborsClassifier` | n_neighbors=5 |
| SVM (Linear) | `sklearn.svm.LinearSVC` | max_iter=5000, random_state=42, dual='auto' |
| XGBoost (Retrained) | `xgboost.XGBClassifier` | random_state=42, eval_metric='logloss' |
| XGBoost (Saved) | Pre-trained pipeline loaded from `saved_model/aml_pipeline.joblib` | Default XGBClassifier parameters; trained on a different under-sample split (no fixed random_state for undersampler) |

### 3.6 Evaluation Metrics

Models were evaluated using the following metrics, computed on the test set (n=3,107):

- **Accuracy**: Overall proportion of correct predictions.
- **Precision** (Class 1): Proportion of predicted laundering transactions that are truly laundering.
- **Recall** (Class 1): Proportion of actual laundering transactions that are correctly detected.
- **F1 Score** (Class 1): Harmonic mean of Precision and Recall for the laundering class.

The F1 score for the positive (laundering) class is treated as the primary evaluation metric, as it balances the trade-off between false positives and false negatives -- both of which carry significant costs in AML applications.

### 4.1 Impact of Class Imbalance on XGBoost

Before applying resampling, a baseline XGBoost model was trained on the original imbalanced dataset (70/30 train-test split, random_state=42). The results demonstrate the failure mode caused by class imbalance:

**XGBoost without Resampling** (test set: 1,521,879 Normal / 1,622 Laundering):

| Laundering (1) | 0.95 | 0.06 | 0.12 | 1,622 |

Despite achieving 0.95 precision for laundering, the model detected only **6% of actual laundering transactions** (recall = 0.06), yielding an F1 score of merely 0.12. ## 5. Discussion

### 5.1 Model Performance Analysis

XGBoost achieved the highest F1 score (0.8892) for laundering detection, but the margin over simpler models is notably small. Logistic Regression, the simplest model tested, achieved an F1 of 0.8825 -- only 0.67 percentage points lower -- while training 38x faster (0.04s vs. 1.53s). This narrow gap suggests that the discriminative signal in the one-hot encoded categorical features is largely linear in nature, and that the non-linear capacity of ensemble methods provides only marginal improvement.

The top five models (XGBoost, Logistic Regression, Random Forest, Decision Tree, SVM) all achieved F1 scores between 0.8729 and 0.8892, a spread of only 1.63 percentage points. This clustering indicates that performance may be approaching a ceiling imposed by the feature representation rather than the modeling algorithm.

KNN was the clear outlier with an F1 of 0.8158 -- approximately 7 percentage points below the best model. This is attributable to the curse of dimensionality: the one-hot encoded feature space has extremely high dimensionality (thousands of features from bank and account identifiers), in which Euclidean distance-based methods degrade.

### 5.2 Saved Model vs. Retrained XGBoost

The saved XGBoost model (F1=0.8639) underperformed the retrained XGBoost (F1=0.8892) by 2.53 percentage points. This discrepancy is explained by a methodological difference: the saved model was trained using `RandomUnderSampler()` without a fixed `random_state`, resulting in a different subset of normal transactions being selected for training. Furthermore, the saved model's `OneHotEncoder` was fit on its own training split, meaning category mappings may differ from the comparison experiment. Despite this, the saved model's performance remains within the competitive range (Rank 6 of 7).

### 5.3 Precision-Recall Trade-offs

An important pattern emerges across models: most classifiers favor **high recall** (0.91-0.94) at the expense of precision (0.83-0.85) for the laundering class. This means they detect the vast majority of laundering transactions but produce some false positives. In practical AML applications, this trade-off is generally acceptable, as the cost of missing a laundering transaction (regulatory penalties, criminal proceeds escaping detection) typically exceeds the cost of investigating a false alarm.

The saved XGBoost model exhibits an interesting reversal: it has higher precision (0.88) but lower recall (0.85) compared to the other models. This suggests that differences in training data composition can meaningfully shift the precision-recall balance.

### 5.4 Feature Representation Limitations

A notable limitation of the current approach is that the `ColumnTransformer` was configured to apply One-Hot Encoding only to 7 categorical columns, with the default `remainder='drop'` setting. This means three potentially informative features were excluded:

- **Account.1** (receiver account identifier): Dropped because it was not listed in the categorical column specification.
- **Amount Received**: A continuous numerical feature that could capture anomalous transaction sizes.
- **Amount Paid**: A continuous numerical feature that could reveal discrepancies between paid and received amounts.

The exclusion of transaction amount features is particularly noteworthy, as amount-based anomaly detection is a cornerstone of traditional AML rule-based systems. Incorporating these features (e.g., via `remainder='passthrough'` in the ColumnTransformer) could potentially improve model performance.

### 5.5 Known Warnings

During both training and inference, scikit-learn's `OneHotEncoder` raised warnings about unknown categories in columns corresponding to `From Bank`, `Account`, and `To Bank`. These arise when the test set contains bank or account identifiers not seen during training -- an expected consequence of under-sampling, which excludes the majority of normal transactions and their associated identifiers. The `handle_unknown='ignore'` setting ensures these unseen categories are encoded as all-zero vectors, preventing errors but potentially losing discriminative information.

## 6. Conclusion

This study demonstrates that machine learning classifiers can effectively detect money laundering transactions in synthetic financial data, provided that class imbalance is properly addressed. Key findings include:

1. **Class imbalance handling is critical**: Without resampling, even XGBoost achieves only 6% recall for laundering transactions. Random Under-Sampling increased this to 94.4%.

2. **XGBoost is the top performer**: With an F1 score of 0.8892 (Precision=0.84, Recall=0.94), XGBoost provides the best balance of detection capability and false alarm rate.

3. **Simple models are competitive**: Logistic Regression (F1=0.8825) performs nearly as well as XGBoost, suggesting that the categorical feature encoding captures most of the discriminative signal through linear decision boundaries.

4. **KNN is unsuitable**: The high-dimensional one-hot encoded feature space degrades KNN performance (F1=0.8158), making it the least effective approach among those tested.

5. **Feature engineering matters**: The current pipeline uses only categorical features. Incorporating numerical features (Amount Received, Amount Paid) and engineering additional features (e.g., transaction frequency, amount ratios, time-of-day patterns) represents a clear avenue for improvement.
